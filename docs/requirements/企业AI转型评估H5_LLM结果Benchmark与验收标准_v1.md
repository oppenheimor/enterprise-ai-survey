## 1. 文档目标

本文档用于建立一套最小可执行 benchmark，防止结果生成陷入“看起来还行”的错觉。

目标是回答四个问题：

1. 输出是否和输入一致。
2. 输出是否和评分结果一致。
3. 输出是否具有基本说服力。
4. 输出是否越界、空泛或自相矛盾。

---

## 2. 验收维度

每条样本按 5 个维度评分，单项满分 20，总分 100。

### 2.1 输入一致性

判断结果是否能回溯到问卷输入。

扣分场景：

1. 忽略用户主场景
2. 忽略用户最困扰问题
3. 忽略预算、负责人、接受度等约束

### 2.2 评分一致性

判断结果是否服从结构化评分结果。

扣分场景：

1. `待提升` 却写成“可以马上全面推进”
2. `top_scene=客服效率` 却推荐从内容生产入手

### 2.3 证据链完整性

判断结果是否说清楚“为什么”。

合格要求：

1. 至少 3 条依据
2. 每条依据可回溯到输入

### 2.4 业务说服力

判断结果是否足以让企业负责人觉得“这个结论和我有关”。

合格表现：

1. 不空泛
2. 不只喊口号
3. 给出能执行的小步建议

### 2.5 边界与风险控制

判断结果是否克制、不越界。

扣分场景：

1. 编造行业数据
2. 编造 ROI 数字
3. 过度承诺

---

## 3. 判定阈值

| 总分 | 判定 |
|------|------|
| 90-100 | 通过，可上线 |
| 80-89 | 基本通过，可灰度 |
| 70-79 | 不稳定，需要继续调优 |
| 0-69 | 不通过，禁止上线 |

同时设置硬阻断：

1. 任何样本出现幻觉事实 -> 直接失败
2. 任何样本出现与评分冲突 -> 直接失败
3. 任何样本无边界说明 -> 直接失败

---

## 4. 样本集设计

最小样本集建议 24 条，按 6 类场景分层，每类 4 条：

1. 高潜力 / 高可行性
2. 高痛点 / 低可行性
3. 低基础 / 高意愿
4. 中型企业 / 流程复杂
5. 服务业 / 客服效率场景
6. 零售或制造 / 内部协同与流程场景

---

## 5. Benchmark 样本模板

每条样本包含：

1. 问卷输入
2. 结构化评分结果
3. 预期主推荐场景
4. 禁止输出项
5. 验收点

模板如下：

```json
{
  "case_id": "case_001",
  "input": {},
  "expected": {
    "top_scene": "",
    "rating": "",
    "must_include": [],
    "must_not_include": [],
    "required_evidence_count": 3
  }
}
```

## 6. 样本示例

### case_001

场景：服务业，中小团队，客服重复工作高，已有少量 AI 基础

期望：

1. 主推荐场景应为 `客服效率`
2. 结论应偏“小范围试点”
3. 必须提到重复性工作和响应速度
4. 不得输出具体收益金额

### case_002

场景：预算低、负责人不明确、团队接受度低

期望：

1. 不得输出“立即全面推进”
2. 必须指出最大阻力是执行闭环不足
3. 建议应偏低风险尝试或先补基础

### case_003

场景：主问题是流程不规范，工具基础弱

期望：

1. 要提升“流程标准化”优先级
2. 不应直接推荐重 AI 能力堆叠
3. 要提示先补工具基础

---

## 7. 验收清单

每次模型、prompt、评分规则变更后，至少检查：

1. 样本通过率是否下降
2. 幻觉率是否上升
3. 套话率是否上升
4. 结果是否更长但更空
5. 推荐场景是否更易漂移

---

## 8. 套话识别规则

以下输出倾向视为“套话风险”：

1. 高频出现“拥抱 AI”“抓住机会”这类空口号
2. 几乎不提用户具体选择
3. 证据部分无法对应输入
4. 不论什么样本都给相似建议

建议对每轮抽样结果额外做“去重审查”，看不同输入下输出是否过于同质化。

---

## 9. 上线前最低要求

1. 24 条 benchmark 样本全部跑通
2. 总体平均分 ≥ 85
3. 不得出现硬阻断项
4. 每类场景至少保留 1 条人工 review 记录
