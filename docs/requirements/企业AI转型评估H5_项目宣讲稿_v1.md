## 宣讲稿

这次项目我们已经把甲方原始模糊需求收敛成了一套相对完整的执行文档，当前主方案只有一条，不要再按旧版本理解。

第一，产品范围先统一。我们现在做的是一个 `企业 AI 转型评估 H5`，核心目标是让企业负责人通过 `16 道选择题` 完成一次初步诊断，系统基于 `规则评分 + LLM 解释层分析` 生成结果预览，再通过留资解锁完整结果。这里有几个边界要特别明确：问卷本体全部是选择题，不要求用户手填业务信息；但留资页的姓名、手机号、企业名称仍然保留文本输入。当前范围内不做复杂 CRM、不做技术型深度诊断、不做精确 ROI 承诺、不做完整实施方案。

第二，给设计师。前台主流程已经固定为 8 个页面：落地页、4 个问卷步骤页、结果预览页、留资页、成功页。同时我又补了一份 `16 题逐屏线框`，所以设计上你们现在要先做一个选择：最终是采用“4 步分组页”表达，还是“16 题逐题页”表达。我的建议偏向逐题页，因为移动端更轻、用户压力更小、埋点也更细。无论选哪种，结果页都必须严格围绕 7 块内容展开：当前阶段判断、核心问题、推荐方向、判断依据、当前阻力、下一步建议、边界说明。不要在结果页塞报价、复杂图表或行业大报告，那会破坏可信度。看板这边也已经补了最小运营看板需求，设计重点是指标卡、漏斗、分布图、线索列表四层结构，不是做花里胡哨的大屏。

第三，给开发。你们不要把这个项目理解成“做个问卷页 + 调个大模型”那么简单。现在已经定义了三层逻辑：第一层是 `16 题问卷字段与埋点`，第二层是 `评分常量表 + 评分映射表`，第三层是 `LLM 提示词框架 + 输出 schema`。正确做法应该是：问卷答案先进入结构化规则引擎，算出分数、等级、主推荐场景、线索等级；然后把问卷答案、结构化评分、约束条件一起传给 LLM，让它只做解释，不做瞎算。前后端必须共享同一份评分常量，不允许各算各的。字段与埋点映射表也已经补齐，包括 LLM 输出字段，所以结果页渲染和落库要按那张表来。再提醒一句，看板虽然不做复杂 CRM，但最小运营看板的数据链路必须现在就考虑，不然后面前台上线了，数据看不到。

第四，给测试。这个项目的关键不是表单有没有提交成功，而是“结果是否自洽、是否越界、是否像套话”。所以测试分三层：第一层是前台流程测试，检查 16 题作答、进度、留资、异常提示、移动端适配；第二层是规则测试，核对相同答案是否稳定得到相同评分、评级和推荐场景；第三层是 LLM 结果测试，这部分我已经补了 `benchmark 与验收标准` 和 `24 条 JSON fixture`。你们后面要重点看几类问题：结果是否和输入一致，是否和评分一致，是否有证据链，是否出现幻觉事实，是否过度承诺，是否不同样本输出过于同质化。只要出现“和评分冲突”“编造行业数据”“编造 ROI”“没有边界说明”，都应该视为阻断问题。

最后说当前文档状态。现在这套文档我重新评估大概是 `96 分`，已经可以作为正式开发基线。剩下没完全收死的点主要有三个：一是最终交互到底走“逐题页”还是“4 步页”；二是评分映射表和评分常量表后续最好合并成唯一配置源；三是 benchmark fixture 还没真正接进代码跑。除此之外，不要再回头按 7 题旧口径理解，也不要再把甲方原始材料里的技术型智能体、投资评分、报价表偷偷带回来。

如果你们内部开启动会，我建议按这个顺序讲：

1. 先讲唯一主方案和范围边界
2. 再讲前台用户流程
3. 再讲评分与 LLM 两段式结果生成
4. 最后讲看板与测试回归标准
