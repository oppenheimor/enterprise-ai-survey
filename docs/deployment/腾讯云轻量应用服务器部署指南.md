## 2. 当前项目部署约束

基于当前仓库实际情况，本项目线上部署至少需要以下运行条件：

1. `Node.js` 运行环境。
2. `PostgreSQL` 数据库。
3. `Prisma migrate deploy` 的执行环境。
4. 一组稳定的环境变量配置。
5. `Nginx` 反向代理与 HTTPS 入口。

当前仓库中的关键约束如下：

1. `Next.js` 版本为 `16.2.4`。
2. `Next.js` 对 Node 的最低要求为 `>= 20.9.0`。
3. 当前项目脚本已经提供：
   - `pnpm build`
   - `pnpm start`
   - `pnpm prisma:deploy`
   - `pnpm check`
4. 当前 `.env.example` 已经定义了数据库和 LLM 相关环境变量。
5. 当前本地数据库通过 `docker-compose.yml` 使用 `5433` 端口，但线上若直接安装 PostgreSQL，默认应使用 `5432`。

因此，当前阶段最适合的线上形态是：

1. `腾讯云 Lighthouse`
2. `Ubuntu + Node.js + PM2 + Nginx + PostgreSQL`
3. 应用与数据库先放在同一台机器上

## 3. 为什么推荐 Lighthouse 而不是 CVM

当前阶段优先推荐 `腾讯云轻量应用服务器 Lighthouse`，不建议一开始就选 `CVM`。

原因如下：

1. 当前项目是标准的轻量 Web 应用，不是复杂分布式系统。
2. 项目当前流量规模和架构复杂度都不需要 `CVM` 的高自由度网络与实例编排能力。
3. `Lighthouse` 的控制台、套餐模式和公网带宽配置更简单，更适合首发阶段。
4. 当前目标是尽快把 H5 问卷、结果分析、留资与落库链路跑通，而不是先搭一个过度设计的云平台。

当前阶段与未来阶段的建议分工如下：

1. 首发阶段：`Lighthouse`
2. 稳定运行后：可将数据库迁移到托管 PostgreSQL
3. 访问量明显增长后：再评估是否迁移到 `CVM` 或容器平台

## 4. 推荐购买方案

### 4.1 地域选择

推荐按目标拆成两种：

1. `中国香港`
   - 优点：通常不需要先完成中国内地备案即可上线。
   - 适合：尽快跑通生产部署、先验证产品链路。
2. `广州 / 上海 / 北京`
   - 优点：更适合长期服务中国内地用户。
   - 注意：若正式对外提供网站服务，通常需要走 ICP 备案。

推荐决策：

1. 想尽快上线：先选 `中国香港`
2. 想长期正式承接内地流量：选 `广州` 或 `上海`，并把备案纳入计划

### 4.2 套餐建议

建议直接购买以下规格：

1. 产品：`轻量应用服务器 Lighthouse`
2. 操作系统：`Ubuntu 24.04 LTS`
3. CPU / 内存：`2 核 4GB`
4. 磁盘：`100GB SSD` 左右
5. 登录方式：`SSH 密钥` 优先
6. 购买周期：`1 年`

不推荐首发直接选 `2 核 2GB`，原因如下：

1. `pnpm install`
2. `pnpm build`
3. `prisma generate`
4. `prisma migrate deploy`

这些步骤在低内存实例上更容易出现构建慢、内存吃紧、需要频繁补 swap 的问题。

1. 先在 `中国香港` 跑通部署链路。

## 12. 低内存机器的补充处理

如果服务器内存偏小，构建阶段可能比较吃紧。可额外配置 swap：

```bash
sudo fallocate -l 4G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
```

即使当前推荐规格为 `2 核 4G`，在构建高峰期保留 swap 依然有价值。

## 15. 配置 Nginx 反向代理

### 15.1 创建站点配置

```bash
sudo rm -f /etc/nginx/sites-enabled/default
sudo nano /etc/nginx/sites-available/enterprise-ai-survey
```

填入以下内容：

```nginx
server {
    listen 80;
    server_name 你的域名 www.你的域名;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

### 15.2 启用站点

```bash
sudo ln -s /etc/nginx/sites-available/enterprise-ai-survey /etc/nginx/sites-enabled/enterprise-ai-survey
sudo nginx -t
sudo systemctl reload nginx
```

此时，Nginx 已经接管外部访问入口，并将请求转发给本地 `3000` 端口的 Next.js 服务。

### 15.3 本次实际生效的 Nginx 配置

本次服务器最终生效的配置如下，可直接作为当前项目的排障基线：

```nginx
server {
    listen 80;
    server_name 43.137.38.67 www.oppenheimor.tech oppenheimor.tech;

    client_max_body_size 20m;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";

        proxy_read_timeout 60s;
        proxy_connect_timeout 60s;
        proxy_send_timeout 60s;
    }
}
```

同时需要确保：

1. `/etc/nginx/sites-enabled/default` 已被移除
2. `/etc/nginx/sites-enabled/enterprise-ai-survey` 已正确软链接到 `sites-available`

否则即使应用已经运行，公网访问也仍可能看到 Ubuntu/Nginx 默认欢迎页。

### 15.4 当前项目子路径部署配置

如果当前项目需要作为单个子应用部署到 `/enterprise-ai-survey`，当前线上已验证通过的 Nginx 配置如下：

```nginx
server {
    listen 80;
    server_name 43.137.38.67 www.oppenheimor.tech oppenheimor.tech;

    location = / {
        return 404;
    }

    location = /enterprise-ai-survey {
        proxy_pass http://127.0.0.1:3000/enterprise-ai-survey;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }

    location ^~ /enterprise-ai-survey/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 60s;
        proxy_connect_timeout 60s;
        proxy_send_timeout 60s;
    }
}
```

注意：

1. 这套配置建立在项目已经启用 `basePath="/enterprise-ai-survey"` 的前提下
2. 不要试图只靠 Nginx 把根路径应用“伪装成子路径”，否则 API、静态资源和页面跳转很容易漂移

## 17. 配置 HTTPS

### 17.1 申请腾讯云 SSL 证书

建议通过腾讯云 SSL 证书服务申请免费 DV 证书，并下载 `Nginx` 版本证书包。

### 17.2 上传证书到服务器

```bash
sudo mkdir -p /etc/nginx/ssl
sudo cp 你的证书_bundle.crt /etc/nginx/ssl/
sudo cp 你的私钥.key /etc/nginx/ssl/
```

### 17.3 更新 Nginx 配置

将原有站点配置改为：

```nginx
server {
    listen 80;
    server_name 你的域名 www.你的域名;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name 你的域名 www.你的域名;

    ssl_certificate /etc/nginx/ssl/你的证书_bundle.crt;
    ssl_certificate_key /etc/nginx/ssl/你的私钥.key;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

### 20.4 PM2 开机自启的真实踩坑

`pm2 startup systemd -u <user> --hp <home>` 生成 systemd 单元后，并不代表一定已经稳定接管。

本次实际踩到的坑是：

1. 先手工启动了 PM2 daemon
2. 再让 systemd 接管
3. 导致 `pm2-ubuntu.service` 因 PID 所属不一致而启动失败

表现通常是：

1. `systemctl is-enabled pm2-ubuntu` 显示 `enabled`
2. 但 `systemctl status pm2-ubuntu` 仍然是 `failed`
3. 日志出现：
   - `New main PID ... does not belong to service`
   - `Can't open PID file ... pm2.pid`

本次最终可用的修复顺序是：

```bash
sudo -u ubuntu pm2 save
sudo -u ubuntu pm2 kill
sudo systemctl reset-failed pm2-ubuntu
sudo systemctl start pm2-ubuntu
systemctl status pm2-ubuntu --no-pager
sudo -u ubuntu pm2 status
```

经验结论：

1. 要么从一开始就让 systemd 启动 PM2
2. 要么在切换为 systemd 托管前，先把旧的手工 PM2 daemon 清干净

否则你会看到“应用在线，但开机自启是假的”这种很烦的半成品状态

### 20.5 子路径部署的真实踩坑

这次项目迁移到 `/enterprise-ai-survey` 时，踩到了几类非常典型的坑：

1. 不能只改 Nginx，不改 Next.js
2. `basePath` 生效依赖重新构建，不能只改运行时配置
3. `next/link` 在开启 `basePath` 后会自动补前缀，手工再拼一次会重复
4. 客户端裸 `fetch("/api/...")` 不会自动补前缀，需要显式处理
5. `revalidatePath` 按路由文件路径工作，不应该拼 `basePath`
6. 直接在 CSS 里写 `/fonts/...` 这类根路径资源，子路径部署时容易失效，最好收口到 `next/font/local`

经验结论：

1. 子路径部署本质上是应用级行为，不只是网关级行为
2. “页面能打开” 不代表路径语义已经完全正确
