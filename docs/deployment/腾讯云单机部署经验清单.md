# 腾讯云单机部署经验清单

## 文档目标

本文档用于沉淀本次部署讨论中值得长期复用的经验，而不是一次性的操作步骤。

目标是让后续类似项目在部署前先回答对问题，而不是先机械执行命令。

---

## 1. 首发阶段优先原则

首发部署阶段最重要的不是“架构看起来高级”，而是：

1. 能上线
2. 能验证用户主流程
3. 能稳定回滚
4. 能在出问题时快速定位

因此，早期项目默认优先：

1. 更简单的云产品
2. 更少的中间层
3. 更清晰的故障边界

---

## 2. 腾讯云产品选型经验

### 2.1 什么时候优先 Lighthouse

满足以下条件时，优先选 `Lighthouse`：

1. 单体应用
2. 轻量 Web 项目
3. 首发阶段
4. 运维经验有限
5. 当前最重要的是尽快上线

### 2.2 什么时候再考虑 CVM

出现以下情况时，再认真评估 `CVM`：

1. 需要复杂 VPC 与安全组管理
2. 需要更多实例类型和高级网络能力
3. 需要中大型分布式部署
4. 流量、团队协作和运维复杂度已经明显提升

结论：

1. `Lighthouse` 适合首发
2. `CVM` 适合扩张期

---

## 3. 地域选择经验

### 3.1 香港地域的价值

若项目当前目标是“尽快上线验证”，`中国香港` 常常是最省阻力的选择。

原因：

1. 可以先绕开备案阻塞
2. 能更快完成真实线上验证
3. 适合先跑通部署流程和业务链路

### 3.2 内地地域的价值

若项目长期明确服务中国内地用户，`广州 / 上海` 更适合长期运行。

但代价是：

1. 要考虑备案
2. 要更早处理域名主体和合规问题

经验结论：

1. 先验证：香港
2. 进正式运营：内地

---

## 4. 服务器规格经验

对于 `Next.js + Node.js + PostgreSQL` 这类项目，首发阶段不建议过度压低内存。

经验建议：

1. 默认从 `2 核 4G` 起步
2. 磁盘默认从 `100GB` 左右起步

原因：

1. Node 项目安装依赖和构建会吃内存
2. Prisma generate / migrate 也会占资源
3. 2G 机器更容易让构建阶段变成不稳定因素

结论：

1. 省一小点机器钱，可能会换来更多构建和排障成本

---

## 5. 单机部署经验

对于首发项目，单机部署通常是合理的。

推荐默认链路：

1. `Nginx` 负责公网入口
2. `Node.js / PM2` 负责应用运行
3. `PostgreSQL` 先与应用同机部署

单机部署的优势：

1. 组件少，故障定位快
2. 便于快速验证数据库与应用链路
3. 便于回滚和快照

单机部署不等于永远不升级，它只是更适合首发阶段。

---

## 6. 安全经验

默认安全策略应尽量保守：

1. 对公网只开放 `22`、`80`、`443`
2. 数据库端口不要暴露公网
3. 应用内部端口不要暴露公网
4. 尽量使用 SSH 密钥登录

经验结论：

1. “先开放着，后面再关” 通常会变成永远没关
2. 首发阶段的最小暴露面，比事后补洞更重要

---

## 7. 环境变量经验

部署失败里最常见的问题之一，不是代码本身，而是环境变量。

后续项目要重点检查：

1. 本地开发端口是否误带到线上
2. 数据库连接串是否仍指向本地习惯配置
3. 第三方 API Key 是否漏配
4. 回调地址、域名、证书路径是否仍是占位值

经验结论：

1. `.env.example` 只能做起点，不能等于线上配置

---

## 8. 部署验证经验

部署完成不等于项目真正可用。

至少要验证三层：

1. 基础设施层：域名、Nginx、HTTPS、进程守护
2. 业务流程层：用户主流程是否真实可走通
3. 数据层：数据库写入是否成功

经验结论：

1. “页面能打开” 只说明最表层没挂
2. 真正的上线验收必须覆盖一次完整业务链路

### 8.1 先用 IP 验收，再谈域名

单机项目部署时，入口问题经常混在一起，导致排障方向跑偏。

更稳的顺序是：

1. 先验证 `127.0.0.1:3000`
2. 再验证服务器公网 IP
3. 最后再验证域名

经验结论：

1. IP 不通，优先看应用、PM2、Nginx
2. IP 已通但域名不通，优先看 DNS、CDN、Cloudflare、备案或证书

不要在“域名未指向本机”时，误以为是项目本身没有部署成功

### 8.2 Git 仓库应成为线上代码源

首发抢通链路时，可以临时用 `rsync` 或 `scp` 上传代码。

但一旦进入可复用部署流程，应该尽快切换为：

```text
本地 commit/push -> GitHub -> 服务器 pull/build/restart
```

原因：

1. 线上版本可以追溯到 commit。
2. 回滚可以基于 Git，而不是靠猜哪个备份目录能用。
3. 多设备或多人协作时，发布入口更统一。
4. 服务器不依赖某一台开发机的临时工作区。

经验结论：

1. 服务器拉不到未提交、未 push 的代码。
2. `.env`、数据库和 Nginx/PM2 配置属于服务器运行态，不应被提交到 Git。
3. 私有仓库推荐使用只读 deploy key，不要把个人 GitHub 私钥放到服务器。
4. 常规发布优先用 `git pull --ff-only`，只有确认部署目录没有服务器侧改动并已备份时，才使用 `git reset --hard`。

### 8.3 自动部署不要越过手动部署边界

当手动的 GitHub 拉取式部署已经稳定后，可以把同一套命令搬进 GitHub Actions。

但自动化不应该改变几个边界：

1. GitHub 是代码源。
2. 服务器负责使用自己的 deploy key 拉代码。
3. GitHub Actions 只负责 SSH 登录服务器并编排部署命令。
4. 生产 `.env` 默认留在服务器，不因为上了 CI/CD 就上传到仓库。
5. 自动部署应先检查服务器工作树是否干净，再执行 `git pull --ff-only`。
6. 部署日志应按阶段输出，便于判断失败在检查、SSH、拉代码、迁移、构建、PM2、Nginx 还是公网入口。

经验结论：

1. 自动部署的价值是减少重复操作，不是扩大权限。
2. 先有清楚的手动流程，再有可靠的 GitHub Actions。
3. `.env` 和业务密钥属于运行环境状态，不等于代码发布产物。

## 9. 运维最小闭环经验

首发部署完成后，至少补齐以下最小闭环：

1. PM2 开机自启
2. Nginx 配置校验
3. 服务器快照
4. 数据库备份
5. 至少一次重启恢复验证

经验结论：

1. 没有备份和恢复验证的上线，只能算“把程序放上去”

### 9.1 PM2 自启不等于真的可恢复

很多人看到以下结果就以为结束了：

1. `pm2 startup` 执行过了
2. `pm2 save` 执行过了
3. `systemctl is-enabled pm2-xxx` 显示 `enabled`

这还不够。

至少还要确认：

1. `systemctl status pm2-xxx` 是 `active`
2. 由 systemd 托管拉起的 PM2 daemon 和应用进程都在线
3. 执行过一次“先 kill 再由 systemd 拉起”的恢复验证

经验结论：

1. 没做接管验证的 PM2 自启，可信度很低
2. 手工启动过 PM2 后再切换 systemd，最容易踩 PID 冲突坑

### 9.2 Nginx 默认站点是首发高频坑

如果外网打开后看到 Ubuntu/Nginx 欢迎页，不要立刻怀疑应用。

优先检查：

1. `/etc/nginx/sites-enabled/default` 是否仍存在
2. 项目站点文件是否真的落到了 `sites-available`
3. `sites-enabled` 里的软链接是否正确

经验结论：

1. 应用在线不代表公网入口已经切换成功
2. Nginx 默认站点没移除时，排障时间会被白白浪费在错误方向上

### 9.3 域名背后可能还有旧代理层

如果域名历史上挂过 Cloudflare、CDN 或其他代理，部署到新机器后经常出现：

1. 域名解析结果不是当前服务器公网 IP
2. 响应头显示请求并没有真正打到你的机器
3. 你以为是 Nginx 配错了，实际上是域名入口还没切过来

经验结论：

1. 先看 `dig`
2. 再看响应头
3. 最后才看应用代码

### 9.4 子路径部署优先改应用，不要先赌 Nginx 技巧

如果一个 Next.js 应用的正式访问路径要从根路径改成 `/<app-name>`，优先顺序应该是：

1. 先让应用自己支持 `basePath`
2. 再让 Nginx 代理这个子路径

不要反过来先靠 Nginx rewrite 做“伪子路径”。

原因：

1. `next/link`、静态资源、API 请求和缓存失效语义不完全等价
2. 只改网关通常会得到一个“首页像是对的，但局部路径全在偷偷漂移”的半正确系统

经验结论：

1. 子路径部署是应用行为 + 网关行为的组合
2. 只改 Nginx，通常只够做临时演示，不够做稳定上线

---

## 10. 架构演进经验

早期项目的正确顺序通常是：

1. 先把业务跑通
2. 再把稳定性补齐
3. 最后再升级架构

不推荐的顺序是：

1. 一开始就追求容器化、多集群、多副本、复杂编排
2. 结果业务还没验证，团队已经先掉进运维泥潭

经验结论：

1. 架构复杂度应该滞后于业务复杂度，而不是领先太多

---

## 11. 后续项目复用方式

后续启动类似项目时，建议按以下顺序复用：

1. 先读本经验清单，判断是否仍适合单机首发
2. 若适合，再用 `腾讯云单机 Web 项目首发部署模板.md`
3. 在项目真正开始部署时，再补一份项目专用部署文档

这样可以把知识分成三层：

1. 原则层：经验清单
2. 模板层：通用部署模板
3. 项目层：项目专用指南
