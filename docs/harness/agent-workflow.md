# 智能体工作流
本项目将大模型视为**执行工作者**，代码仓库视为**管控框架**。
在追求功能开发效率之前，管控框架必须先保障三件事：

1. 稳定的指令规范
2. 稳定的任务文档
3. 稳定的验证机制

---

## 层级对应关系

### 规则层
- `AGENTS.md`：项目整体定位与特殊约束、执行规则

### 规范层
- `openspec/changes/*`：进行中的变更需求
- `openspec/specs/*`：已确认的基准行为规范

### 验证层
- `scripts/check.sh`：默认本地验证入口脚本
- package.json 脚本：供人员与智能体使用的稳定命令别名

---

## 何时必须使用 OpenSpec
若工作涉及以下内容，**编码前必须创建或更新 OpenSpec 变更文档**：
- 面向用户的调研流程
- 评分或分析逻辑
- 数据模型与持久化方案
- 接口结构
- 外部系统集成
- 非简单的 UI 行为

仅修改文案、或孤立的小 bug 修复，且不改变预期行为时，可不创建新的变更文档。

## 任务编写标准
一份合格的任务或变更应明确：
- 目标
- 范围
- 非目标（不做什么）
- 验收标准
- 验证方式

若缺少任意一项，智能体将通过猜测补全，这通常是产生劣质代码修改的根源。

## 验证标准
默认本地验证命令：
```bash
pnpm check
```
若任务需要额外校验，需将补充命令写入当前 OpenSpec 任务或实现说明中。
