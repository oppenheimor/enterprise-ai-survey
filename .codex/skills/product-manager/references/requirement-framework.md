# 需求分析方法论与框架

## 一、需求分析五步法

### 1. 场景还原
**目的**: 理解用户在什么场景下遇到问题

**方法**:
- 用户访谈
- 实地观察
- 日志分析
- 问卷调查

**输出**:
- 用户场景描述
- 使用环境分析
- 触发条件梳理

### 2. 痛点挖掘
**目的**: 找到问题的根本原因

**工具**:
- 5Why分析法
- 鱼骨图
- 用户旅程地图

**示例**:
```
问题: 用户投诉搜索功能不好用
Why 1: 搜索结果不准确
Why 2: 搜索算法没有考虑用户意图
Why 3: 缺少用户行为数据分析
Why 4: 没有埋点收集用户搜索行为
Why 5: 产品规划时未考虑数据驱动

根因: 产品设计阶段缺乏数据思维
```

### 3. 价值评估
**目的**: 判断解决问题的价值大小

**评估维度**:
| 维度 | 问题 |
|------|------|
| 用户价值 | 解决了多少用户的痛点？ |
| 业务价值 | 对业务指标有什么影响？ |
| 战略价值 | 是否符合产品战略方向？ |
| 竞争价值 | 能否形成差异化优势？ |

### 4. 方案设计
**目的**: 提出可行的解决方案

**设计原则**:
- MVP优先: 先做最小可行方案
- 用户中心: 以用户体验为导向
- 技术可行: 考虑实现难度
- 成本可控: 评估资源投入

### 5. 优先排序
**目的**: 确定实施顺序

**排序方法**:

#### Kano模型
| 类型 | 定义 | 策略 |
|------|------|------|
| 基本型 | 必须有，没有会不满 | 优先保证 |
| 期望型 | 越多越好 | 按需投入 |
| 魅力型 | 意外惊喜 | 适度投入 |
| 无差异 | 有无都行 | 不投入 |
| 反向型 | 越多越不满 | 避免 |

#### MoSCoW法则
| 类型 | 定义 | 占比建议 |
|------|------|----------|
| Must have | 必须有 | 60% |
| Should have | 应该有 | 20% |
| Could have | 可以有 | 15% |
| Won't have | 暂不做 | 5% |

## 二、需求优先级评估矩阵

### ICE评分法

```
ICE得分 = Impact(影响) × Confidence(信心) ÷ Effort(投入)
```

| 维度 | 评分标准 (1-10) |
|------|-----------------|
| Impact | 1=影响很小，10=影响巨大 |
| Confidence | 1=不确定，10=非常确定 |
| Effort | 1=投入很小，10=投入巨大 |

### RICE评分法

```
RICE得分 = Reach(覆盖) × Impact(影响) × Confidence(信心) ÷ Effort(投入)
```

## 四、竞品分析框架

### 分析维度

| 维度 | 分析内容 |
|------|----------|
| 产品定位 | 目标用户、核心价值、差异化 |
| 功能对比 | 功能列表、功能深度、功能创新 |
| 用户体验 | 交互设计、视觉设计、易用性 |
| 商业模式 | 盈利模式、定价策略、渠道 |
| 技术架构 | 技术栈、性能、稳定性 |
| 运营策略 | 推广渠道、用户增长、内容运营 |

### SWOT分析模板

| | 有利 | 不利 |
|------|------|------|
| **内部** | Strengths (优势) | Weaknesses (劣势) |
| **外部** | Opportunities (机会) | Threats (威胁) |

## 五、产品规划方法

### 版本规划

```
v1.0 MVP版本 - 核心功能
├── 用户注册登录
├── 核心业务流程
└── 基础数据展示

v1.1 优化版本 - 体验优化
├── 性能优化
├── 交互优化
└── Bug修复

v1.2 增强版本 - 功能扩展
├── 高级功能
├── 数据分析
└── 系统集成

v2.0 重大版本 - 战略升级
├── 架构升级
├── 新业务线
└── 平台化
```

### OKR拆解

```
Objective: 提升用户活跃度
├── KR1: DAU从1万提升到3万
├── KR2: 用户留存率从30%提升到50%
└── KR3: 人均使用时长从5分钟提升到10分钟
```

## 六、需求文档规范

### 文档结构

1. **背景与目标** - 为什么做，要达成什么目标
2. **目标用户** - 为谁做，用户画像
3. **功能需求** - 做什么，功能详细说明
4. **非功能需求** - 性能、安全、兼容性
5. **数据需求** - 埋点、存储、统计
6. **风险评估** - 可能的问题和应对
7. **项目排期** - 时间计划和里程碑

### 验收标准编写

**好的验收标准**:
- 具体明确，可测试
- 覆盖正常和异常场景
- 使用Given-When-Then格式

**示例**:
```
✅ Given 用户已登录 When 点击"提交"按钮 Then 系统保存数据并显示成功提示
❌ 用户可以提交数据（太模糊，不可测试）
```
