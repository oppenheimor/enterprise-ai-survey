## Context

项目当前的结果链路建立在“点击结果预览后立即生成完整模型报告”的前提上，并且把结果页、留资页和成功页写成同一个状态机。这个结构有两个问题：

1. 它把本地规则、模型诊断和页面展示耦合成了一个入口。
2. 它默认预览结果必须继承旧的 `scoring + diagnosis` 契约，难以支持新的角色化预览。

这次改动属于入口级重构，需要从“Q16 点按钮之后发生什么”重新定义流程，而不是在旧数据结构上补丁式迁移。

## Goals / Non-Goals

**Goals:**

- 建立独立的本地评估核心层，统一生成 6 维分数、阶段、综合指数和角色匹配结果。
- 用新的预览结果对象驱动结果页展示，避免页面直接消费评分规则或模型报告 schema。
- 将前台主流程收敛为落地页、问卷页和结果预览页。
- 保留模型诊断和留资代码资产，但使其不再参与当前结果入口。

**Non-Goals:**

- 不在本轮恢复详细分析入口。
- 不在本轮删除模型诊断、图表和留资落库代码。
- 不修改数据库结构和后台看板。
- 不重写 16 题问卷结构或题目内容。

## Decisions

- **以领域层重构取代页面层打补丁**：新增 `assessment` 模块，形成 `SurveyAnswersInput -> AssessmentResult -> PreviewResult` 的明确层次。替代方案是复用旧 `PreviewState` 并替换字段，但会继续让模型报告语义污染本地预览。
- **先判阶段，再判角色**：阶段作为高一层判断稳定用户感知，再在候选角色池中做强命中与相似度排序。替代方案是 16 个角色直接平铺匹配，但边界 case 容易漂移。
- **评分、阶段、角色目录都配置化**：将题目映射、维度权重、阶段规则和角色元数据集中管理，避免逻辑散落在多个函数和组件中。替代方案是把规则写死在引擎函数里，但后续调整成本高且难以回归。
- **结果页只消费投影后的预览对象**：页面组件只关心角色头图、维度条和文案，不接触原始问卷推导细节。替代方案是页面直接读领域结果，但会逐步长回逻辑页。
- **主流程直接移除留资和成功状态**：不是隐藏按钮，而是让当前链路没有半成品分支。替代方案是保留状态但不展示入口，这会让前端状态机仍然背着旧结构。
- **保留模型诊断入口作为未来详细分析网关**：`lib/survey/diagnosis.ts` 和相关测试继续保留，但不在当前 action 中触发。替代方案是直接删除，但后续恢复会增加重复劳动。

## Risks / Trade-offs

- **规则量上升，维护复杂度增加** → 用集中配置、显式命中依据和测试样本约束，避免黑盒 if/else。
- **旧 3 维评分和新 6 维评估并存易让命名混乱** → 新评估模块使用新的类型和文件夹，不复用旧 `scoring` 结果对象。
- **移除留资页后，旧代码路径可能失活** → 保留模块但从主流程解耦，同时用注释说明是未来详细分析资产。
- **前端结果页重做后，旧图表组件暂时闲置** → 本轮不强制删除，减少无谓改动面。
