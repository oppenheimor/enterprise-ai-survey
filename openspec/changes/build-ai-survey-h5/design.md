## Context

`docs/requirements` 已经把产品范围压缩得比较清楚：这是一个服务企业负责人的移动端 H5，不是 ROI 计算器，不是技术测评系统，也不是完整 CRM。当前技术仓库仍处于脚手架阶段，但需求层已经同时定义了前台页面、16 题问卷、评分映射、字段与埋点、LLM 输出边界、benchmark 样本和最小运营看板，所以这次设计必须先把跨页面流程与跨模块边界钉牢。

本次设计还要解决两个容易把项目带偏的问题。第一，问卷交互在文档里同时存在“4 步分组表达”和“16 题逐屏线框”；结合 `企业AI转型评估H5_16题逐屏线框_v1.md` 与 `企业AI转型评估H5_项目宣讲稿_v1.md` 的推荐，本 change 默认采用 `16 题逐题页`，同时保留分组标题用于认知分段。第二，UI 方向已经由用户指定为 `Bento Grid`，因此要把它落成真正服务移动端转化的视觉系统，而不是继续停留在风格展示页面。

## Goals / Non-Goals

**Goals:**
- 以单一主流程交付可上线的 H5：落地页、16 题逐题问卷、结果预览、留资解锁、成功页。
- 用共享配置驱动问卷题目、选项值、评分常量、推荐场景和埋点字段，避免前后端各算各的。
- 把结果生成拆成两段：规则评分负责判断，LLM 负责解释，并通过固定 schema 约束输出。
- 在首轮设计中就考虑持久化和运营看板，避免前台跑起来后数据链路断掉。
- 让验证与实现顺序对齐：先能约束高风险逻辑，再补强 UI 与交互回归。

**Non-Goals:**
- 不实现复杂 CRM、顾问协同工作台或线索自动分配系统。
- 不交付行业 benchmark 报告、精确 ROI 预测或报价能力。
- 不把 Bento Grid 做成独立风格选择系统或 A/B 风格投票系统。
- 不把 4 步分组页和 16 题逐题页同时做成双模态正式功能。

## Decisions

### 采用“16 题逐题页 + 分组标题”的问卷交互
- 决策：正式用户流程采用逐题页，每屏一题，顶部展示总进度与当前题号，同时保留“企业情况 / 当前痛点 / 基础与条件 / 决策偏好”的分组标题。
- 原因：需求文档已经明确指出移动端逐题模式更轻、压力更低、埋点更细，也更利于中断恢复与主流程完成率优化。
- 备选方案：保留 4 步分组页作为正式实现。放弃原因是单页题目过多会拉长滚动面，削弱移动端节奏控制，也不利于题级埋点与校验。

### 用共享问卷配置作为前后端唯一事实源
- 决策：把 16 题定义、选项值、分组信息、评分映射、评级区间、场景映射和线索等级规则收敛到一组共享配置模块，由前端渲染与服务端计算共同消费。
- 原因：宣讲稿和评分文档已经明确要求不能“各算各的”。共享配置能同时降低 drift 风险，并为规则测试和 fixture 回归提供统一输入。
- 备选方案：前端只负责展示，后端单独维护评分常量。放弃原因是任何一侧更新都可能造成埋点、显示文案、选项值与评分不一致。

### 结果生成采用“规则引擎 + LLM 解释”双层结构
- 决策：先由确定性规则产出 `score_total`、`score_rating`、`top_scene`、`lead_priority` 与维度分，再把问卷答案、结构化结果和约束条件传给 LLM 生成结构化 JSON 输出。
- 原因：需求文档已经把“LLM 不能自己发明评分和事实”写得很死。双层结构能让模型只在自己该发光的地方发光，别顺便客串算命先生。
- 备选方案：直接把答案喂给 LLM，让模型自行判断评分和建议。放弃原因是不可验证、不可回归，也无法满足 benchmark 的硬阻断要求。

### 前台提交与运营查询共享同一份持久化模型
- 决策：提交完成后统一保存问卷答案、评分结果、LLM 输出摘要、留资信息与关键埋点字段，并让最小运营看板直接从同一持久化层查询汇总。
- 原因：PRD 与最小运营看板需求都强调“线索与问卷结果完整落库”。如果前台和看板分两套存储，项目很快会出现“页面能看、运营看不到”的荒诞现实。
- 备选方案：先只做前台流程，后续再补后台。放弃原因是这会把数据字段设计变成二次迁移任务，届时最容易重写 schema 和补历史数据。

### Bento Grid 作为视觉系统，而不是额外页面
- 决策：把 Bento Grid 体现在卡片布局、层级编排、信息块组合、结果页信息分区和移动端节奏上，而不是新增一个仅供展示的风格页。
- 原因：仓库里已有 `add-style-wall-showcase` 完成风格探索，本次 change 的职责是把风格用于正式用户路径。
- 备选方案：继续沿用现有简单表单风格，Bento Grid 留待后续。放弃原因是用户已经明确 UI 风格，且落地页与结果页的专业感直接影响转化。

## Risks / Trade-offs

- [一次性覆盖前台、评分、LLM、落库、看板，范围偏大] → 通过 OpenSpec tasks 分阶段落地，先打通主流程，再接规则与结果，再补运营看板。
- [Bento Grid 在移动端容易做成“卡片很多但重点不明”] → 限制每屏信息密度，优先突出 CTA、进度、当前问题和结果结论，不做横向炫技拼贴。
- [共享配置既承担渲染又承担计算，修改时可能牵一发动全身] → 用显式 schema 和规则测试保护配置，禁止在页面组件内散落魔法值。
- [LLM 输出不稳定，容易出现套话或越界] → 强制结构化输入输出、接入 benchmark 样本，并把“与评分冲突 / 编造事实 / 无边界说明”视为阻断。
- [最小运营看板需求容易被随手做成“先能看再说”的临时页] → 在规格里限定只做指标卡、漏斗、分布图和线索列表四层，不扩成 CRM。

## Migration Plan

- 第一步：补齐共享问卷配置、评分规则、结果 schema 与 Prisma 数据模型。
- 第二步：实现前台主流程页面与提交链路，先确保不依赖 LLM 也能完成规则评分与持久化。
- 第三步：接入 LLM 解释层与结果页渲染，同时加入 benchmark 验证。
- 第四步：实现最小运营看板，并基于同一批数据做查询与筛选。
- 回滚策略：若 LLM 或看板部分不稳定，可保留前台问卷、规则评分与留资主链路上线，暂时关闭解释层或后台入口，但不得破坏已落库数据结构。

## Open Questions

- LLM 调用将采用哪一种服务端封装方式，目前需求文档定义了输入输出协议，但未固定具体 provider 接口。
- 结果页“预览内容”与“留资后完整内容”的差异边界还需要在实现前进一步收紧，避免前后内容只是换个按钮颜色。
- 最小运营看板是否需要基础筛选条件以外的导出能力，当前文档未要求，本 change 默认不纳入范围。
