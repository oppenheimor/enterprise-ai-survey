## Context

项目当前已经具备规则评分、Vercel AI SDK 模型调用和结果预览页。但诊断输出 schema 只有 9 个短字段，前端展示也只有分数、三项维度数字、依据和下一步建议，无法支撑用户对“专业系统分析”的期待。

本次设计要把“结果摘要”升级为“AI 转型体检报告”。关键不是无限加长文案，而是让报告结构可视化、可回溯、可解锁，并继续遵守不编造 ROI、不输出未经验证行业事实的边界。

## Goals / Non-Goals

**Goals:**
- 在点击“查看结果预览”时生成完整结构化报告。
- 预览页展示足够有价值的核心结论、维度体检和场景建议。
- 留资成功后展示完整报告内容，而不是只显示提交成功。
- 报告包含图表渲染数据、工具类型建议和阶段行动路线。
- 保留规则评分作为模型分析锚点，避免模型自行发明分数和评级。

**Non-Goals:**
- 不做实时流式报告生成动画。
- 不联网抓取最新工具榜单，不让模型随口推荐具体品牌事实。
- 不承诺 ROI、节省百分比、行业 benchmark 或完整实施方案。
- 不重写问卷题目和评分规则。

## Decisions

- **一次生成，分层展示**：点击查看结果时生成完整报告，预览页只展示 `preview` 层，留资后展示 `fullReport` 层。替代方案是预览和完整报告分别生成，但会带来结论漂移和两次模型成本。
- **schema 以报告模块建模**：不保留旧 9 字段摘要契约，`diagnosis` 直接升级为 `dimensionReports`、`visualAnalysis`、`toolRecommendations`、`actionRoadmap` 等完整报告结构。这样前端可稳定渲染图表和模块，不依赖模型自由文本。
- **工具推荐只推荐类型与用法**：先输出“智能客服/知识库问答/流程自动化”等工具类别、适用场景、落地方式和注意事项。具体品牌推荐需要未来维护工具库后再做，否则容易幻觉。
- **fallback 也输出完整报告**：模型失败时不能回到短摘要，否则体验质量会随机塌陷。fallback 必须按同一 schema 生成规则化报告。
- **数据库只保存完整诊断 JSON**：删除旧诊断摘要列，避免长期双写和字段漂移。运营看板继续使用评分、场景和线索字段，诊断详情统一读 `diagnosis` JSON。

## Risks / Trade-offs

- 报告字段增多导致模型更容易输出无效结构 -> 使用 Zod schema 强校验，失败走完整 fallback。
- 报告变长可能压低移动端阅读效率 -> 预览页只展示关键模块，完整内容留到解锁后展开。
- 工具建议可能被误解为采购清单 -> 文案明确为“工具类型/使用方式”，不输出未维护的品牌事实。
- 数据库迁移涉及已有表 -> 直接迁移到唯一 `diagnosis` JSON 字段，避免兼容债；开发库数据可重建或按迁移脚本处理。
