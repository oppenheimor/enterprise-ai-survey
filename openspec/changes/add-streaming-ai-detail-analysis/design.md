## Context

项目当前已经有两块资产可以复用：

1. 本地评估驱动的结果预览页
2. 现有模型诊断逻辑与图表组件

但两者之间还缺一条完整的“解锁 AI 深度分析”链路。现状的问题不是模型能不能调，而是：

1. 留资入口已经被从主流程摘出
2. 详情分析页不存在
3. 模型输出没有流式体验，长耗时会造成空白等待
4. prompt 输入还不够完整，没有显式包含问题列表和全部答案文本

这次设计要补的是单页 H5 中的第二阶段体验：在本地预览之后，通过留资弹窗解锁 AI 全面分析，并用流式详情页体现模型能力。

## Goals / Non-Goals

**Goals:**

- 在预览页增加 `AI 全面分析` 按钮和留资弹窗。
- 留资成功后进入单页 `detailAnalysis` 状态。
- 详情分析页支持机器人头部、图表区和流式 AI 文本输出。
- prompt 输入显式包含 16 题问题列表、用户答案、本地评估结果和输出边界。
- 保持图表先展示、文本后流式追加，避免长时间空白等待。

**Non-Goals:**

- 不新建独立详情分析路由页面。
- 不修改数据库 schema。
- 不重写本地评估引擎。
- 不实现完整 AGUI 事件协议，只做当前需求足够的流式分层展示。

## Decisions

- **继续使用单页状态机而非独立路由**：在 `SurveyApp` 中新增 `detailAnalysis` 主状态和 `leadModalOpen` 局部状态。替代方案是跳转独立 `/analysis` 页面，但会增加跨页状态传递与 H5 断裂感。
- **留资用弹窗而不是整页**：复用现有 `leadFormSchema` 和提交逻辑，减少主流程重型页面回归。替代方案是恢复整页留资，但会把本次重构重新拉回旧流程。
- **图表先展示，文本流式输出**：详情页加载后立即展示机器人头部和图表，本地评估与结构化结果作为稳定内容，模型 narrative 通过文本流式追加。替代方案是等模型全部返回再渲染整页，但等待成本太高。
- **流式文本走 route handler，不走 server action**：route handler 更适合持续输出增量文本，前端也更容易按流读取。替代方案是继续用 server action，但很难稳定做文本流。
- **prompt 输入升级为四层上下文**：问题列表、用户答案、本地评估、输出边界同时传入，避免模型只围绕摘要评分空转。替代方案是继续只传评分摘要，但解释深度和对题目上下文的贴合度不足。
- **结构化结果与流式 narrative 分层**：最终结构化诊断继续保留以支撑图表和落库，流式 narrative 作为过程体验层。替代方案是直接把旧 `diagnosisSchema` 拿来做流式，但前后端都很难消费。

## Risks / Trade-offs

- **前端状态机复杂度增加** → 保持单页状态只有一个新增主状态和一个弹窗状态，不恢复旧 success 页。
- **模型流式输出与最终结构化结果可能不同步** → 明确图表只读本地/结构化数据，流式 narrative 只做解释，不改分数。
- **详情分析页内容过长导致移动端阅读压力大** → 将流式文本拆成模块逐段出现，而不是一整段长文。
- **留资提交与流式调用连续发生，失败路径复杂** → 提交成功才切详情页；模型流失败时保留图表和错误提示，不丢失页面上下文。
